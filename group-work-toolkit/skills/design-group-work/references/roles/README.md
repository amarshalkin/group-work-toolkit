# Роли проектирования групповой работы

Подкаталог описывает, **под какую роль** проектируется сессия. Роль определяет угол зрения, язык артефактов и набор речевых заготовок. Одна и та же сессия, спроектированная «под модератора», «под организатора» и «под заказчика», даёт разные документы.

## Текущий состав

- `moderator.md` — **основная рабочая роль v0.10.** Модератор ведёт сессию. Инструкция и Карта результатов адресованы ему. Файл — связка: ссылка на `../moderator-roles.md` (шесть ролей модератора) + правила того, как роль модератора влияет на язык артефактов.
- `organizer.md` — **заглушка**. Организатор отвечает за площадку, логистику, материалы. В v0.10 пересекается с `../org-rider-checklist.md`, но отдельного языка артефактов ещё нет.
- `client.md` — **заглушка**. Заказчик — лицо, принимающее решение о сессии и потребляющее её результат. В v0.10 не проработана.

## Как добавить новую роль

1. Создай файл `<role-name>.md` по шаблону ниже.
2. Определи, какой артефакт адресуется этой роли (или создай новый шаблон в `../../assets/`).
3. Опиши, чем язык артефакта для этой роли отличается от языка для модератора.
4. Добавь evals-кейс, где итоговый артефакт — для этой роли.

### Шаблон файла роли

```markdown
# <Название роли>

## Кто это
Одно-два предложения: кто такой, за что отвечает на сессии и вокруг неё.

## Что получает на выходе
Какой артефакт адресован этой роли. Либо ссылка на существующий шаблон, либо описание нового.

## Отличия в языке
Чем язык артефакта для этой роли отличается от языка «для модератора».
(Пример: заказчику не нужны речевые заготовки, нужны метрики успеха.)

## Какие разделы обязательны
Минимальный набор блоков в артефакте под эту роль.

## Типичные вопросы роли
3–5 вопросов, на которые артефакт должен отвечать прямо.
```

## Правила совместимости

- Роль **не заменяет** методологию. Это ортогональные оси. Одна и та же сессия в КСП может проектироваться и под модератора, и под заказчика — получатся разные артефакты, но методологическая основа одна.
- В `SKILL.md` (Шаг 2) роль модератора уточняется как часть вводных. Если v0.11+ расширит переключатель ролей — нужно отразить это в Шаге 2.
