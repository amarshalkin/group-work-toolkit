# Пост-сессионный разбор (debrief)

Скилл входит в плагин `group-work-toolkit` и может быть явно вызван через `/group-work-toolkit:debrief`. Закрывает петлю: проект → проведение → разбор → улучшение плагина. Выходы скилла — это вход для iteration-2 плагина.

## Когда запускать

- Сессия уже прошла, у пользователя есть наблюдения модератора, заметки, фотографии флипчартов, заполненные раздатки, отзывы участников.
- Пользователь хочет понять, что сработало и что нет, и хочет зафиксировать это, а не забыть.
- Нужно сформулировать улучшения для методички / шаблонов / чек-листов.

## Что скилл делает

1. Собирает у пользователя три источника: (а) Карту ожидаемых результатов (что планировалось), (б) наблюдения модератора и участников (что реально произошло), (в) финальные артефакты группы (что группа произвела).
2. Сопоставляет план с фактом по каждому такту.
3. Выделяет: что сработало, что не сработало, где были сюрпризы.
4. Извлекает выводы двух уровней: **для методички** (речевые заготовки, схемы, формы работы, которые надо добавить/убрать) и **для плагина** (что изменить в `design-group-work`, `review-session-design`, в шаблонах).
5. Выдаёт итоговый отчёт по шаблону `assets/debrief-report-template.md`.

## Основной процесс

### Шаг 1. Собери входные данные

Задай пользователю через интерактивный механизм вопросов:

1. **Карта результатов сессии** — путь к файлу или цитата. Если нет Карты — это уже диагноз, дальше работай с предположением «Карты не было, восстановим по факту».
2. **Наблюдения модератора** — что сам модератор заметил: где группа зависла, где выскочила, какие речевые заготовки сработали, какие — нет, какие страховочные ходы пригодились.
3. **Голоса участников** — была ли быстрая рефлексия в конце, отзывы, анкеты. Если нет — пропусти этот источник, но отметь как allergy в отчёте.
4. **Артефакты группы** — что реально произвела группа: тексты, списки, схемы, голосования. Минимум — перечень с короткими описаниями.

Не требуй всё. Минимум — Карта + наблюдения модератора. Остальное — если есть.

### Шаг 2. Сопоставь план с фактом

Для каждого такта из Карты результатов:

- **Планировалось:** <цитата из Карты>.
- **Получилось:** <факт из наблюдений / артефактов>.
- **Диагноз:** совпало / частично / не совпало / вышло больше запланированного.

Не оценивай в терминах «хорошо/плохо». Факт первичен.

### Шаг 3. Вычлени паттерны

Пройди по трём осям:

1. **Процесс:** где группа зависла, где пошла быстрее плана, где потребовалась импровизация модератора.
2. **Содержание:** какие темы «выстрелили», какие не зашли, какие трактовки объекта/предмета/рамки пришлось менять на лету.
3. **Роль модератора:** где сработала конкретная роль (Капитан / Переводчик / Катализатор / Сталкер / Иной / Просветитель), где пришлось переключаться.

### Шаг 4. Сформулируй выводы двух уровней

**Уровень 1 — для методички/следующей сессии.** Что знать при планировании следующего похожего события: какую схему взять, какие речевые заготовки обязательны, какой тайминг реалистичен.

**Уровень 2 — для плагина group-work-toolkit.** Конкретные правки в артефактах плагина:

- Что добавить в `design-group-work/SKILL.md`.
- Какие речевые заготовки усилить в `assets/moderator-brief-template.md`.
- Какие новые «страшилки» добавить в `references/scary-stories.md`.
- Какие пункты уточнить в `references/design-review-checklist.md` (вход для `review-session-design`).
- Нужен ли новый evals-кейс.

Выводы второго уровня — это **основной ценностный продукт скилла**. Они идут в iteration-2.

### Шаг 5. Собери отчёт по шаблону

Используй `assets/debrief-report-template.md`. Выдай готовый документ. Предложи сохранить в `skills/group-work-design-workspace/` рядом с iteration-1.

## Что важно помнить

- **Не ищи виноватых.** Debrief — про учёбу, не про оценку. Если модератор где-то промахнулся, формулируй как «пункт для усиления методички», а не «модератор сделал ошибку».
- **Факт → интерпретация.** Сначала зафиксируй, что произошло. Только потом объясняй, почему и что с этим делать. Не наоборот.
- **Выводы должны быть actionable.** «Надо больше рефлексии» — не вывод. «Добавить в такт 2 рефлексивный круг 10 минут перед сборкой, потому что группа ушла в раздражение без него» — вывод.
- **Сюрпризы важнее провалов.** Места, где группа выскочила за рамки запланированного и произвела неожиданно хорошее — ценнее провалов. Именно оттуда растут новые схемы и формы.
- **Debrief не замена рефлексии в сессии.** В нормальной сессии последний такт — рефлексия с участниками. Debrief — это то, что делает **модератор + заказчик + методолог** после, уже на холодную голову.

## Связь с другими скиллами

- **design-group-work** — предыдущий шаг цикла. Debrief проверяет его качество на практике.
- **review-session-design** — если в debrief всплыли ошибки, которые должен был ловить чек-лист, — это сигнал обновить `design-review-checklist.md`.
